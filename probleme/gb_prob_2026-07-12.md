## 1. Kurzfazit

Die App hat mehrere kritische und hohe Probleme. Besonders kritisch sind Mandantentrennung, Account-Erstellung/Rechte, OktoPOS/Oktopus-Konfiguration und wirtschaftliche Datenkonsistenz.

Die wichtigsten Probleme:

1. Admins koennen potentiell Config-Daten fremder Organisationen schreiben.
2. Schicht-Cloud-Functions koennen ueber manipuliertes Payload in fremde Organisationen schreiben.
3. OktoPOS/Oktopus `baseUrl` ist nur auf `https://` begrenzt und kann API-Keys an fremde Server senden.
4. Neue Nutzer koennen bei bestimmten Invite-Daten ihre Permissions eskalieren.
5. Neue Nutzer koennen bei Account-Erstellung abrechnungsrelevante Werte manipulieren.
6. Produktbearbeitung kann aktuellen Bestand ueberschreiben.
7. Inventur kann bei altem UI-Stand falsche Bestaende erzeugen.
8. Umlagerung zwischen Standorten ist nicht atomar.
9. Kassenabschluss kann im Hybrid-Modus als gebucht markiert werden, obwohl Journalbuchung nur lokal gespeichert wurde.

## 3. Kritische Probleme

### K1 - Cross-Tenant-Write auf Org-Config

Schweregrad: Kritisch

Fundstellen:

- `firestore.rules:639-642`
- `firestore.rules:76-78`
- `lib/services/firestore_service.dart:157-195`

Problem:

`organizations/{orgId}/config/{configId}` erlaubt Writes fuer jeden Admin. Die Regel prueft nicht, ob der Admin zur Ziel-Organisation `orgId` gehoert.

Auswirkung:

Ein Admin aus Organisation A kann potentiell Config-Dokumente von Organisation B schreiben. Besonders gefaehrlich ist das fuer `config/oktoposSync`, weil dort OktoPOS/Oktopus-Parameter wie `baseUrl`, `enabled` und Standortkonfiguration liegen.

Betroffene Bereiche:

- Mandantentrennung
- Oktopus/OktoPOS
- Feature Flags
- Remote-Konfiguration

### K2 - Cross-Tenant-Write auf Organisationsdokumente

Schweregrad: Kritisch

Fundstellen:

- `firestore.rules:631-634`

Problem:

`organizations/{orgId}` erlaubt `create` und `update` fuer jeden Admin, solange `request.resource.data.name` ein String ist. Es fehlt die Bindung an die eigene Organisation.

Auswirkung:

Ein Admin einer Organisation kann Organisationsdaten einer anderen Organisation anlegen oder veraendern. Das ist ein direkter Bruch der Mandantentrennung.

Betroffene Bereiche:

- Mandantentrennung
- Organisationsverwaltung
- Stammdaten

### K3 - Schicht-Cloud-Functions koennen in fremde Organisationen schreiben

Schweregrad: Kritisch

Fundstellen:

- `functions/index.js:1040-1073`
- `functions/index.js:1076-1112`
- `functions/index.js:2765-2767`
- `functions/index.js:3916`

Problem:

Die Callables `upsertShiftBatch` und `publishShiftBatch` pruefen zwar das Top-Level-`orgId`, aber `parseShift` uebernimmt ein im Payload enthaltenes `org_id`. `writeShiftBatch` schreibt anschliessend mit Admin SDK in `shifts[0].orgId`.

Auswirkung:

Ein berechtigter Planer aus Organisation A kann mit manipuliertem Payload Schichten in Organisation B schreiben. Firestore Rules schuetzen hier nicht, weil Admin SDK verwendet wird.

Betroffene Bereiche:

- Mandantentrennung
- Schichtplanung
- Cloud Functions
- Admin-SDK-Schreibpfade

### K4 - OktoPOS/Oktopus API-Key kann an fremde HTTPS-Server gesendet werden

Schweregrad: Kritisch

Fundstellen:

- `functions/index.js:4695-4702`
- `functions/index.js:5685-5698`
- `functions/index.js:5308-5323`
- `lib/services/firestore_service.dart:157-195`

Problem:

Die OktoPOS/Oktopus `baseUrl` wird serverseitig nur darauf geprueft, ob sie mit `https://` beginnt. Es gibt keine Host-Allowlist, keine Origin-Pruefung und keinen Schutz gegen fremde HTTPS-Domains.

Auswirkung:

Wenn ein Angreifer die `baseUrl` beeinflusst, sendet `oktoposFetch` den Header `X-API-KEY` an einen fremden Server. In Kombination mit K1 kann ein Admin einer fremden Organisation die OktoPOS-Config manipulieren und API-Keys exfiltrieren.

Betroffene Bereiche:

- Oktopus/OktoPOS
- Secrets
- Cloud Functions
- SSRF/Secret Exfiltration

### K5 - Account-Erstellung erlaubt Permission-Eskalation bei Invites ohne Permissions

Schweregrad: Kritisch

Fundstellen:

- `firestore.rules:522-537`

Problem:

Beim Erstellen eines `users/{uid}`-Dokuments wird `permissionsEquivalent` nur verlangt, wenn die eigene Einladung ein Feld `permissions` enthaelt. Wenn eine Einladung keine `permissions` hat, kann der Nutzer im Create-Payload eigene Permissions setzen.

Auswirkung:

Ein neuer Nutzer kann sich bei Legacy- oder manuell fehlerhaften Einladungen zusaetzliche Rechte geben, z. B. Inventar-, Kontakt-, Schicht- oder Admin-nahe Rechte.

Betroffene Bereiche:

- Login
- Account-Anlage
- Rechte/Rollen
- Firestore Rules

### K6 - Account-Erstellung schuetzt Lohn- und Urlaubsfelder nicht ausreichend

Schweregrad: Kritisch

Fundstellen:

- `firestore.rules:522-537`
- `firestore.rules:557-560`

Problem:

Beim Update sind Payroll-Felder wie `settings.hourlyRate` und `settings.vacationDays` geschuetzt. Beim initialen Create des Nutzerprofils fehlt dieser Schutz.

Auswirkung:

Ein neuer Nutzer kann beim ersten Profil-Write potentiell eigene Lohn- oder Urlaubswerte setzen. Das ist abrechnungsrelevant.

Betroffene Bereiche:

- Login
- Account-Anlage
- Personal
- Abrechnung

## 4. Hohe Probleme

### H1 - userInvites sind fuer fremde Admins sichtbar

Schweregrad: Hoch

Fundstellen:

- `firestore.rules:589-596`

Problem:

`userInvites` erlaubt `get` und `list` fuer jeden Admin, ohne die Organisation des Invite-Dokuments gegen die Organisation des Admins zu pruefen.

Auswirkung:

Admins koennen Einladungen anderer Organisationen sehen. Das betrifft E-Mail-Adressen, Rollen und Organisationszuordnung.

Betroffene Bereiche:

- Datenschutz
- Account-Anlage
- Mandantentrennung

### H2 - OktoPOS/Oktopus siteId ist nicht ausreichend validiert

Schweregrad: Hoch

Fundstellen:

- `functions/index.js:4494-4498`
- `functions/index.js:4660-4688`
- `functions/index.js:4702-4705`

Problem:

Die `siteId` kommt aus dem Request. Es wird nicht streng geprueft, ob diese Site in der OktoPOS-Config existiert und zur Organisation gehoert. Gleichzeitig erlaubt die API-Key-Aufloesung Fallbacks auf `*` oder `default`.

Auswirkung:

OktoPOS/Oktopus-Syncs oder Pushes koennen fuer falsche oder unbekannte Sites laufen. Bei Wildcard-/Default-Keys kann trotzdem ein echter API-Key verwendet werden.

Betroffene Bereiche:

- Oktopus/OktoPOS
- Standorttrennung
- Secrets
- Kassendaten

### H3 - OktoPOS/Oktopus Bewegungs-ID kann bei fehlender Line-ID kollidieren

Schweregrad: Hoch

Fundstellen:

- `functions/index.js:4883-4886`
- `functions/index.js:5047-5053`
- `functions/index.js:5450-5457`

Problem:

Die Bewegungs-ID nutzt `item.id`. Wenn OktoPOS/Oktopus fuer mehrere Zeilen keine `item.id` liefert, wird `0` verwendet. Mehrere Belegzeilen koennen dadurch dieselbe `movementId` erhalten und werden dedupliziert.

Auswirkung:

Nicht alle Verkaufszeilen werden als Bestandsbewegung gebucht. Der Bestand wird zu hoch angezeigt, Wareneinsatz und Auswertungen werden falsch.

Betroffene Bereiche:

- Oktopus/OktoPOS
- Warenwirtschaft
- Bestand
- Kassenimport

### H4 - OktoPOS/Oktopus Pagination kann nach erster Seite abbrechen

Schweregrad: Hoch

Fundstellen:

- `functions/index.js:4769-4774`
- `functions/index.js:4917`
- `functions/index.js:4919-4930`

Problem:

Wenn die API kein `lastPage` liefert, faellt `pageLastPage` auf die aktuelle Seite zurueck. Die Schleife kann nach Seite 1 enden.

Auswirkung:

Nur ein Teil der Transaktionen wird verarbeitet. Der Cursor kann trotzdem fortgeschrieben werden, wodurch nicht gelesene Daten dauerhaft fehlen.

Betroffene Bereiche:

- Oktopus/OktoPOS
- Kassenimport
- Umsatzdaten
- Bestandsbewegungen

### H5 - OktoPOS/Oktopus Page-Cap kann Datenverlust verursachen

Schweregrad: Hoch

Fundstellen:

- `functions/index.js:4917`
- `functions/index.js:4919-4930`

Problem:

Die Sync-Schleife endet bei `OKTOPOS_MAX_PAGES`. Danach kann der Cursor trotzdem fortgeschrieben werden.

Auswirkung:

Wenn mehr Seiten vorhanden sind als erlaubt, bleiben Transaktionen ungelesen. Durch den fortgeschriebenen Cursor koennen diese Transaktionen dauerhaft ausgelassen werden.

Betroffene Bereiche:

- Oktopus/OktoPOS
- Kassenimport
- Umsatzdaten
- Bestandsdaten

Receipt- und Movement-IDs enthalten `cashRegisterId` oder Fallbacks wie `all`/`siteId`. Wenn sich die Konfiguration aendert, entstehen fuer dieselben Belege andere IDs.

Auswirkung:

### H8 - Produktbearbeitung kann aktuellen Bestand ueberschreiben

Schweregrad: Hoch

Fundstellen:

- `lib/providers/inventory_provider.dart:1044-1059`
- `lib/repositories/firestore_inventory_repository.dart:348-359`
- `lib/models/product.dart:256-259`

Problem:

Beim Speichern eines Produkts wird `fridgeStock` geschuetzt, aber `currentStock` bleibt im Firestore-Payload. Ein alter Produktstand aus der UI kann einen neueren serverseitigen Bestand ueberschreiben.

Auswirkung:

Verkaufsabgaenge, Inventuren oder Wareneingaenge koennen durch spaetere Produktstammdaten-Edits rueckgaengig gemacht werden.

Betroffene Bereiche:

- Warenwirtschaft
- Bestand
- Produktpflege
- Oktopus/OktoPOS-Bestandsabgleich

### H9 - Inventur berechnet Delta aus potentiellem stale UI-Bestand

Schweregrad: Hoch

Fundstellen:

- `lib/providers/inventory_provider.dart:2451-2465`
- `lib/repositories/firestore_inventory_repository.dart:464-486`

Problem:

`recordStocktake` berechnet `delta = countedStock - product.currentStock` im Provider. Der `product.currentStock` kann veraltet sein. Die Repository-Transaktion liest zwar frisch, addiert aber das bereits falsch berechnete Delta.

Auswirkung:

Eine Inventur kann auf einem falschen Endbestand landen. Beispiel: UI zeigt 10, Server ist 9, gezaehlt wird 7. Delta ist -3, Server landet bei 6 statt 7.

Betroffene Bereiche:

- Warenwirtschaft
- Inventur
- Bestand

### H10 - Umlagerung zwischen Standorten ist nicht atomar

Schweregrad: Hoch

Fundstellen:

- `lib/providers/inventory_provider.dart:2572-2624`

Problem:

Eine Umlagerung besteht aus zwei getrennten Bestandsbuchungen: Abgang Quelle, Zugang Ziel. Wenn die Zielbuchung scheitert, wird nur best-effort kompensiert.

Auswirkung:

Bestand kann verschwinden oder Bewegungslogs koennen inkonsistent werden. Bei wirtschaftlichen Daten ist das ein ernstes Konsistenzproblem.

Betroffene Bereiche:

- Warenwirtschaft
- Standortbestand
- Audit-Log

### H11 - Kassenabschluss kann falsch als zu Finance gebucht markiert werden

Schweregrad: Hoch

Fundstellen:

- `lib/providers/finance_provider.dart:533-556`
- `lib/providers/finance_provider.dart:724-785`
- `lib/screens/daily_closing_screen.dart:243-269`

Problem:

`postDailyClosing` speichert Journalbuchungen ueber `saveJournalEntry`. Im Hybrid-Modus kann `_persist` bei Firestore-Fehlern lokal speichern. Danach kann der Kassenabschluss trotzdem in der Cloud als `bookedToFinance=true` markiert werden.

Auswirkung:

Die App zeigt einen Kassenabschluss als gebucht, obwohl die Journalbuchung nicht in der Cloud existiert. Das ist fuer Buchhaltung, DATEV und Reporting gefaehrlich.

Betroffene Bereiche:

- Wirtschaftsbereich
- Kasse
- Finance
- Tagesabschluss

## 6. Niedrigere oder operative Probleme

### N1 - Public Wishes/Feedback sind ohne App-Check-Enforcement missbrauchbar

Schweregrad: Niedrig bis Mittel, abhaengig von Produktivkonfiguration

Problem:

Oeffentliche oder halb-oeffentliche Eingaben sind nur dann gut gegen Abuse geschuetzt, wenn App Check und ggf. Rate Limits produktiv korrekt aktiv sind.

Auswirkung:

Ohne Enforcement koennen Spam, Kosten und Datenmuell entstehen.

Betroffene Bereiche:

- Public Input
- Firestore Kosten
- Betriebssicherheit

### N2 - CSP ist als Meta-Tag gesetzt, nicht als HTTP-Header

Schweregrad: Niedrig bis Mittel

Fundstellen:

- `web/index.html`
- `firebase.json`

Problem:

Die Content Security Policy steht im HTML-Meta-Tag. In `firebase.json` sind mehrere Security Header gesetzt, aber kein HTTP-`Content-Security-Policy` Header.

Auswirkung:

Ein HTTP-CSP-Header ist robuster. Bei XSS waere das wegen Firebase Auth Persistence besonders relevant.

Betroffene Bereiche:

- Web
- Security Headers
- XSS-Haertung

### N3 - Analyzer meldet bestehende Hinweise/Warnungen

Schweregrad: Niedrig

Fundstellen:

- `analysis_options.yaml:37`
- `lib/screens/home_screen_tabs.dart:2235`
- `lib/screens/search/global_search.dart:405`

Problem:

`flutter analyze` meldet drei bestehende Issues:

- Entfernte Lint-Regel `avoid_returning_null_for_future`.
- Ungenutzter optionaler Parameter `onNavigateBack`.
- Deprecated API `announce`.

Auswirkung:

Kein akuter Produktivfehler, aber CI-/Qualitaetsrauschen und kuenftige Wartungsprobleme.

Betroffene Bereiche:

- Codequalitaet
- Wartbarkeit

### N4 - Viele Dependencies sind deutlich hinter aktuellen Versionen

Schweregrad: Niedrig bis Mittel

Problem:

Beim Aufloesen der Flutter-Abhaengigkeiten wurden viele neuere Paketversionen gemeldet, darunter Firebase-, FlutterFire- und Plattformpakete.

Auswirkung:

Veraltete Pakete koennen Sicherheits-, Stabilitaets- oder Kompatibilitaetsprobleme enthalten. Gleichzeitig ist ein grosses Upgrade riskant und muss separat getestet werden.

Betroffene Bereiche:

- Dependencies
- Firebase
- Langfristige Wartbarkeit
